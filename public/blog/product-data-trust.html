Product Data for AI Agents: Why Extraction Isn't the Hard Part

Krishna Brown · March 24, 2026

I'm building a personal, very niche hybrid agentic commerce platform as a way to learn how to build with AI. The platform uses agents to find, compare, and purchase products on my behalf. It started as a weekend project and turned into something much more instructive than I expected.

The first thing I needed was product data. Not curated feeds or affiliate APIs, but real-time structured data from any product page on the open web. So I built an extraction engine. It parsed Schema.org markup when available, fell back to LLM-based extraction when it wasn't, and returned clean JSON for every URL I threw at it.

Extraction worked. Within a few days I had structured product data flowing through my agents. But I immediately hit a wall that had nothing to do with parsing HTML or prompting language models. The wall was trust. My agents were making decisions (adding items to carts, comparing prices, triggering purchase flows) based on data they couldn't evaluate. A price field extracted by an LLM from a cluttered product page carried the same weight as a price pulled directly from Schema.org markup. My agents treated both as equally reliable, and they were not.

LLMs returned wrong prices. They hallucinated availability. They confused variant titles with product names. Not often, but often enough that an autonomous agent acting on that data would make costly mistakes. The extraction was working. The trust layer was missing.

From any URL to scored, structured product data

That experience became the foundation of ShopGraph. The core idea: extraction is a solved problem in most cases. Schema.org gives you structured data when it exists. LLMs can extract from raw HTML when it doesn't. The unsolved problem is telling the consumer of that data how much to trust each field.

ShopGraph returns per-field confidence scores alongside every extraction. A price pulled from Schema.org markup gets a baseline confidence of 0.93. The same field extracted by an LLM from unstructured HTML might score 0.70. When both sources agree, the hybrid score rises to 0.85 or higher. The agent consuming this data can then decide: is 0.70 confidence on a price good enough to show a comparison, or should it flag the result for human review before adding to a cart?

This is not a cosmetic addition. It changes what agents can do. Without confidence scores, agents either trust everything (and make mistakes) or trust nothing (and require constant human oversight). With field-level confidence, agents can be selective. They can trust the brand name at 0.95 while flagging a price at 0.68 for verification. They can set thresholds per field, per use case, per risk tolerance.

What I learned immediately

The first thing I learned is that Schema.org adoption is uneven. Large retailers tend to have thorough structured data markup. Smaller shops, marketplaces, and niche categories often have partial or outdated Schema.org. This means any production system needs the LLM fallback, and that fallback needs to be clearly labeled as lower confidence.

The second thing: field-level granularity matters more than document-level scoring. A product page might have perfect Schema.org for the name and brand but completely miss availability or shipping details. A single confidence score for the whole extraction hides exactly the information an agent needs to make decisions.

The third: agents need the score at call time, not as a separate lookup. If you have to make a second API call to check confidence, most integrations won't bother. The score has to ship inline with the data, in the same response, in the same object. That's why ShopGraph returns confidence as a sibling field to every extracted value.

What the market is telling me

I started building ShopGraph for myself. The commerce platform is still running, still using ShopGraph as its data layer. But as I talked to other developers building agent-based systems, a pattern emerged: everyone has an extraction step, and everyone hits the same trust problem.

Travel agents pulling hotel prices. Research agents comparing B2B supplier specs. Shopping assistants aggregating deals across dozens of retailers. The domain varies but the problem is identical. The data comes back structured, but the agent has no way to know which fields to trust.

The conversations that surprised me most were with teams building B2B procurement agents. In B2B, a wrong price or an incorrect MOQ (minimum order quantity) has real financial consequences. These teams weren't just interested in confidence scores as a nice-to-have. They needed them as a hard requirement before their agents could operate autonomously.

What's next

ShopGraph is live today as a REST API and an MCP server. You send a product URL (or raw HTML), and you get back structured product data with per-field confidence scores. The output is compatible with the Universal Commerce Protocol spec, so it plugs directly into agent frameworks that speak UCP.

The playground is the fastest way to see it in action. Paste any product URL and you'll get the full scored response.

I'm currently working on an AgentReady scoring system that evaluates product data across multiple dimensions: structured data completeness, semantic richness, UCP compatibility, pricing clarity, inventory signal quality, and access readiness. The goal is to give a holistic view of how well a product page serves agent-based consumers, not just whether the data can be extracted.

If you're building agents that consume product data, I'd like to hear what trust problems you're running into. The patterns I've seen so far suggest this is a shared infrastructure problem, not something every team should solve independently.
