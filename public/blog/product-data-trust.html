Product Data for AI Agents: Why Extraction Isn't the Hard Part

Krishna Brown · March 24, 2026

Your agent can extract product data from any URL in a few lines of code. Point an LLM at a product page, ask for the price, title, and availability. You'll get something back. But you won't know if any of it is right.

The LLM will confidently return a wrong price. It will hallucinate availability. It will confuse a variant title with the product name. If a procurement agent trusts that data and issues a purchase order, that's a real cost. Not a hypothetical. A line item on someone's books.

Extraction is a solved problem. Trust is not. This isn't unique to commerce. Any system where an agent acts on externally sourced data faces the same question: how confident should you be in what came back, and does that confidence vary by field? ShopGraph solves this for product data: extract from any commerce URL, score every field for confidence, and return structured JSON where your agent knows exactly how much to trust each data point.

How it works

ShopGraph uses a tiered extraction strategy. Instead of sending every page through the same pipeline, it tries the cheapest and most reliable method first, then falls back only when needed. Each tier carries a different confidence baseline, so the score your agent receives reflects not just what was extracted but how it was extracted.

The three tiers work as a waterfall. Each one only fires if the previous tier couldn't produce a complete result.

Tier 1: Schema.org markup. Parse structured data directly from the page. Most major retailers embed product metadata in JSON-LD or microdata format. When this markup exists and covers the fields you need, it's the fastest and most reliable source. Confidence scores from this tier start at 0.90 or higher.

Tier 2: LLM extraction. When structured markup isn't there, or when it's incomplete, ShopGraph sends the page content to a language model with a structured extraction prompt. The model returns typed fields, but because LLMs can misinterpret ambiguous layouts or confuse similar-looking elements, confidence scores from this tier are lower, typically in the 0.50 to 0.75 range.

Tier 3: Headless browser rendering. When the page requires JavaScript execution to display product data, ShopGraph renders it in a headless browser before attempting extraction. This is the slowest and most expensive tier, and confidence varies depending on the complexity of the rendered page.

The system tries the cheapest, most reliable method first and only escalates when it has to. This keeps latency low and confidence high for the majority of requests, since most major commerce sites do have structured markup for core product fields.

The scoring layer

Every field in the response carries a confidence score between 0 and 1. The score reflects both the extraction method and the internal consistency of the result. A price parsed from Schema.org JSON-LD scores differently than the same price extracted by an LLM from a page that also displays sale prices, bundle prices, and per-unit prices in close proximity.

This per-field granularity is what makes the system useful for agents. A product page might have perfect structured markup for the name and brand but no availability signal at all. A single document-level confidence score hides that distinction. Field-level scoring exposes it, so your agent can trust the brand at 0.97 while treating availability at 0.55 as provisional.

Without confidence scores, agents either trust everything (and make mistakes) or trust nothing (and require constant human oversight). With field-level scoring, agents can be selective. They can set thresholds per field, per use case, per risk tolerance. A comparison tool might accept 0.60 on price. A procurement agent issuing purchase orders might require 0.90. What I learned building this

I listed ShopGraph on every MCP directory I could find: Glama, Smithery, mcpservers.org, MCP Market, the Official MCP Registry. Zero usage. Not low usage. Zero.

Even if they had driven traffic, there is a paradigm mismatch. MCP is currently indexed toward desktop chat clients like Claude Desktop and Cursor. The B2B data teams who need product data infrastructure build headless, server-side pipelines. They don't install MCP servers. They call REST APIs.

The REST API had to become the primary integration surface. The MCP server now acts as a wrapper around the same engine for developers already in that ecosystem. The playground, where someone can paste a URL and see results without signing up, turned out to be the real conversion path. Directory listings just confirm you exist.

B2B is harder and more valuable than consumer use cases. In consumer shopping, a wrong price is annoying. In B2B procurement, a wrong price or an incorrect MOQ (minimum order quantity) is a line item on a purchase order. These teams don't treat confidence scores as a nice-to-have. They need them as a hard requirement before their agents can operate autonomously.

What the market is telling me

Google's Universal Commerce Protocol, Stripe's Merchant Product Pages, Shopify's AI Toolkit. The infrastructure layer for agent-to-commerce interaction is forming. These aren't competing with ShopGraph. They're validating the problem space. If every major platform is building structured product interfaces for AI, the need for a trust layer on top of that data only grows.
