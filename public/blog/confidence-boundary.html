When Should Your AI Agent Trust the Data? Designing the Confidence Boundary

Krishna Brown · April 12, 2026

ShopGraph extracts product data from any URL, scores how much to trust each field, and returns structured JSON. That's what it does. The interesting part is the decisions underneath.

Every product is a set of tradeoffs. The ones that matter most aren't feature choices. They're the constraints you accept and the failure modes you design for. Here are five decisions that shaped how the system handles trust, uncertainty, and the boundary between autonomous action and human judgment.

1. Per-field scoring, not per-document

A document-level confidence score is useless for agent routing. "This extraction is 80% reliable" tells you nothing actionable. The price might be perfect while the availability is a guess. Lumping them together means either the agent treats the guess as reliable or throws away the good data too.

ShopGraph scores every field independently. A response might come back with price at 0.93, brand at 0.94, availability at 0.72, and description at 0.48. Those aren't the same quality of data. The system doesn't pretend they are.

This is the decision that makes confidence routing actually work. A procurement agent can act on the price immediately, flag the availability for human verification, and drop the description entirely. Each field carries its own trust signal. The agent makes a different decision for each one. Try it in the playground and look at the field_confidence object in any response.

2. Cheapest and most reliable first

Why not use an LLM for everything? Because structured markup, when it exists, is both more reliable and cheaper than LLM extraction. The LLM is a fallback, not the primary path.

ShopGraph runs a three-tier extraction pipeline. Tier 1 parses Schema.org markup directly from the page. This is the fastest path, under a second, and the most reliable because the merchant put that data there intentionally. Tier 2 uses LLM extraction when structured markup is missing or incomplete. Slower, more expensive, but handles unstructured pages. Tier 3 renders the page in a headless browser when the initial request gets blocked or the page requires JavaScript execution.

The system tries the cheapest, most reliable method first and escalates only on failure. That's a cost-reliability tradeoff that shapes the entire architecture. A Shopify store with clean Schema.org markup hits Tier 1 and returns in under a second. A B2B industrial portal with a JavaScript-rendered catalog might escalate to Tier 3. Your agent doesn't need to know which path succeeded. But the metadata tells you: every response includes the extraction method and tier so your downstream systems can factor that into their decisions.

3. Design for the developer who skips the safety layer

The assumption that drove strict_confidence_threshold is simple: developers will ignore confidence metadata. They'll grab the price field and never check the score. Every API that returns optional quality signals has this problem. The signals are there. Nobody reads them.

So the system enforces it. Set strict_confidence_threshold in your API request, and fields below that threshold come back as null with an explanation of why. The data was there. The system extracted it. But the confidence didn't meet your bar, so you don't get it.

This is a server-side enforcement mechanism for a client-side discipline problem. If your agent is autonomous and you can't ensure it checks confidence scores before acting, set the threshold. You'll get fewer fields back, but every field you get is above your quality bar.

The tradeoff is real. Threshold filtering removes your ability to route low-confidence fields to human review. You lose visibility into what was removed and why. That's why there are two distinct usage modes: confidence routing for human-in-the-loop workflows (don't set the threshold, evaluate each field yourself) and threshold filtering for fully autonomous pipelines (set the threshold, trust what comes back). The two modes exist because the two use cases have fundamentally different risk profiles.

4. What happens when 0.92 is wrong

A procurement agent trusts a price at 0.92 confidence and issues a purchase order. But the price was wrong. The Schema.org markup was stale. The merchant updated the price on the page but not in the structured data. That's not just bad data. That's a purchase order against a wrong price. A line item on someone's books. An inventory commitment based on false information. If the order is large enough, it's a creditor dispute.

This is the hardest problem in the pipeline. High confidence doesn't mean correct. It means the extraction method and the source data aligned in a way that the scoring system evaluated as reliable. If the source data itself is stale or wrong, the confidence score reflects the quality of the extraction, not the truth of the data. The system can be confident and still wrong. Designing for that failure mode is the difference between a tool and production infrastructure.

How does the pipeline recover? The self-healing architecture re-extracts on a schedule and compares results across tiers. If Tier 1 (Schema.org) returns a price of $49.99 but Tier 2 (LLM extraction from the rendered page) returns $59.99, the system flags a discrepancy. The confidence on that field drops. Downstream agents that previously trusted that field at 0.92 will now see it at a lower score on the next extraction.

This doesn't prevent the first bad purchase order. It prevents the second one. The AgentReady scoring system also tracks how frequently a URL's data changes and how consistent the data is across extraction methods. A URL where Schema.org markup frequently diverges from rendered content gets a lower AgentReady score, which signals to your agent that this source requires more caution before committing resources.

The honest answer is that confident-but-wrong is an unsolved problem across the industry. No extraction system can verify that the data on a commerce site is current. What the system can do is make the uncertainty visible, give you the provenance of every field, and update its confidence when evidence of staleness appears.

5. The threshold is the product decision

The 0.85 confidence threshold that appears in the documentation isn't a technical constant. It's a design choice about acceptable risk.

Lower it to 0.70 and your agent processes more fields autonomously. Faster throughput, broader coverage, more errors. Raise it to 0.95 and almost everything gets flagged for human review. Slower, more expensive, fewer mistakes.

The threshold IS the product decision. It encodes your organization's risk tolerance into a single number. A price comparison tool for consumers might run at 0.70 because a wrong price is an inconvenience, not a financial loss. A procurement system issuing purchase orders against industrial suppliers might run at 0.95 because a wrong price on a bulk order is a $20,000 mistake.

ShopGraph doesn't pick your threshold. It gives you the data to pick it yourself: per-field confidence scores, extraction method provenance, and AgentReady signals about the source URL's data quality. You set the bar. The system shows you exactly where each field falls relative to that bar.

The boundary between what the agent handles and what the human handles is the design problem at the center of every system where automation and human judgment intersect. Confidence scoring is how you draw that boundary with data instead of guessing.
